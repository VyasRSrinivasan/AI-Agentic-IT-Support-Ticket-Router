# Schemas Architecture Details

## Design Philosophy

The system is designed as a decision pipeline, NOT a chatbot.
Agents come up with structured outputs and the workflow/verifier gives constraints as to what actions are acceptable.

## Pipeline

### 1. Ticket 
(_ticket.py_)

- Normalized representation of an incoming support request
- Standardizes ticket data from various sources (email, chat, etc.) into a consistent format
- Includes fields like ticket_id, subject, body, product, and metadata for downstream processing
- Provides a text_for_llm() method to generate canonical text for agent inputs

### 2. DetectorOutput 
(_detector.py_)

- Early safety/completeness signals
- Assesses ticket for potential security risks (e.g., high-risk keywords)
- Checks for missing information that could prevent safe resolution
- Provides flags like security_risk and missing_info to guide routing decisions

### 3. TriageOutput 
(_triage.py_)

- Classification along multiple axes
- Categorizes tickets by department (e.g., Billing, Technical Support)
- Assesses urgency (e.g., High, Medium, Low) and complexity (e.g., Simple, Complex)
- Includes a confidence score to indicate reliability of the classification

### 4. EvidenceBundle 
(_evidence.py_)

- Structured container for retrieved grounding evidence.
- Holds lists of `EvidenceChunk` items 
- Organizes evidence from knowledge base retrieval into a bundle for resolver use
- Includes methods to access all chunks or filter by source for citation tracking 

### 5. ResolutionDraft 
(_resolution.py_)

- Proposed automated response generated by the resolver
- Contains the suggested response text, citations from evidence, and next steps
- Includes a confidence score to indicate how reliable the draft is
- Serves as input to the verifier for safety and grounding checks

### 6. VerificationResult 
(_verifier.py_)

- Safety gate for automation
- Evaluates whether an auto-resolution is safe based on evidence grounding and risk factors
- Provides a boolean is_safe_to_auto_resolve flag and a list of reasons for the decision
- Can override decisions to ensure only low-risk, well-grounded actions proceed

### 7. Decision 
(_decision.py_)

- Final operational action
- Specifies the outcome: AUTO_RESOLVE, ESCALATE, or ASK_CLARIFYING
- Includes routing details like target_queue, escalation_level, and confidence score
- For clarification requests, provides specific questions to ask the user
